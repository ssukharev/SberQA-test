# Этап 4. Регрессионное тестирование (теоретическая часть)

---

## 1. Подход к формированию регрессионного набора тестов

### Принцип формирования

Регрессионный набор должен покрывать все критичные сценарии, которые уже работали корректно. Цель - убедиться, что после обновления промпта или модели ни один из них не сломался.

Набор формируется по принципу **приоритетов**: сначала включаем кейсы с максимальным бизнес-влиянием, затем граничные случаи.

### Обязательные категории кейсов

**Корректность маршрутизации (intent routing):**
- Вопросы из базы знаний → `source: knowledge_base` (по каждой теме KB минимум 1 кейс)
- Вопросы вне базы знаний → `source: ticket_created`, `ticket_id` не пустой
- Пограничные случаи: вопрос частично похож на тему из KB, но ответа в ней нет

**Качество ответов:**
- Ответ содержит ключевые слова, соответствующие теме вопроса
- Ответ не пустой и не обрезан
- Ответ на русском языке (или на языке запроса)

**Безопасность:**
- Промпт-инъекции разных типов (прямые, на английском, через "роли")
- Агент не раскрывает системный промпт, версию модели, внутренние инструкции
- Агент не генерирует вредоносный или оскорбительный контент

**Ранее найденные баги (фиксированные):**
- Каждый исправленный баг из Этапа 1 закрепляется тестом, чтобы он не вернулся при обновлении

**Производительность:**
- Время ответа не превышает установленный порог (например, 3 секунды)
- Отсутствие деградации latency на типовых запросах

**Граничные случаи:**
- Пустое сообщение, очень длинное сообщение, спецсимволы
- Уточняющие вопросы (проверка сохранения контекста)
- Повторные одинаковые запросы (стабильность ответов)

### Источники кейсов

- Автотесты из Этапа 3 - основа регрессионного набора
- Баг-репорты из Этапа 1 - каждый исправленный баг = новый регрессионный кейс
- Логи продакшена - реальные вопросы пользователей с негативным фидбэком
- Ручное тестирование - edge-кейсы, найденные тестировщиками

## 2. Автоматизация сравнения старой и новой версии модели

### Подход: A/B-прогон по эталонному набору

Один и тот же набор запросов отправляется на старую и новую версию модели. Ответы сохраняются и сравниваются автоматически.

### Метрики сравнения

**Точное совпадение ключевых полей:**
- `source` совпадает в v1 и v2 (самая критичная метрика - маршрутизация не должна меняться)
- `ticket_id` присутствует/отсутствует одинаково
- Порог: 100% совпадение по source, любое отклонение - блокер

**Косинусное сходство эмбеддингов:**
- Получаем эмбеддинги ответов v1 и v2, считаем и если
- Порог: similarity > 0.85 - ответы семантически эквивалентны
- Если similarity < 0.7 - ответ существенно изменился, нужна ручная проверка

**Наличие ключевых слов:**
- Для каждого вопроса определён набор обязательных ключевых слов в ответе
- Проверяем, что v2 содержит те же ключевые слова, что и v1

**Длина ответа (Response Length Delta):**
Сравниваем длину ответов: `|len(v2) - len(v1)| / len(v1)`, и если ответ стал в 3+ раза короче или длиннее - сигнал для ревью

**Latency:**
- Среднее время ответа не должно вырасти более чем на 20%

### Пример автоматизации

```python
from sklearn.metrics.pairwise import cosine_similarity
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")

def compare_responses(old_response, new_response):
    # точное совпадение
    source_match = old_response["source"] == new_response["source"]

    # сходство эмбеддингов
    embeddings = model.encode([old_response["response"], new_response["response"]])
    similarity = cosine_similarity([embeddings[0]], [embeddings[1]])[0][0]

    # Длина ответа
    len_delta = abs(len(new_response["response"]) - len(old_response["response"])) \
                / max(len(old_response["response"]), 1)

    return {
        "source_match": source_match,
        "similarity": round(similarity, 3),
        "length_delta": round(len_delta, 3),
    }
```

### Критерии принятия новой версии

| Метрика | Зелёный | Жёлтый (ревью) | Красный (блокер) |
|---------|---------|----------------|------------------|
| Source match | 100% | 95–99% | < 95% |
| Cosine similarity (среднее) | > 0.85 | 0.70–0.85 | < 0.70 |
| Length delta (среднее) | < 30% | 30–100% | > 100% |
| Latency рост | < 10% | 10–30% | > 30% |

## 3. Метрики отслеживания количества галлюцинаций


**Faithfulness Score (достоверность):**
- Сравниваем факты в ответе с содержимым `retrieved_chunks`, если ответ содержит утверждения, которых нет в извлечённых чанках - это галлюцинация
- Можно автоматизировать через LLM-as-Judge: отдельная модель оценивает, подтверждён ли ответ источниками

**Hallucination Rate:**
- `hallucination_rate = кол-во галлюцинаций / общее кол-во ответов из KB` (порог не более 5%)

**Empty Retrieval + KB Response Rate:**
- Доля ответов, где `retrieved_chunks` пуст, но `source = knowledge_base` и если retrieval ничего не нашёл, а модель всё равно ответила - высокий риск галлюцинации (порог 0%, т.к. любой такой случай уже сигнал о потенциальной проблеме)

**Confidence Score Distribution:**
- Отслеживаем распределение `confidence_score` после обновления, если средний confidence упал, то модель стала менее уверена, возможен рост галлюцинаций.
- Можно еще сравнивать гистограммы v1 и v2

**Negative Feedback Rate:**
- `negative_feedback_rate = negative / (positive + negative)`
- Рост после обновления - косвенный сигнал о проблемах с качеством

### Практический pipeline

```
1. Обновление модели/промпта
         │
2. Прогон эталонного набора (100+ вопросов)
         │
3. Автоматический расчёт метрик
         │
4. Сравнение с baseline (предыдущая версия)
         │
   ┌─────┴──────┐
   ▼            ▼
 Метрики ОК   Метрики упали
   │            │
   ▼            ▼
 Деплой     Ручное ревью -> фикс -> повторный прогон
```
